## 二、架构优化建议

### 2.1 🔴 P0 - URL/Quiz 任务应迁移到 API

**当前问题**：Python 项目用浏览器方式执行 URL/Quiz 任务，效率低且不稳定。

**TS 项目 API 方式**：

```
POST https://www.bing.com/bingqa/ReportActivity?ajaxreq=1
POST https://rewards.bing.com/api/reportactivity
```

**优势**：

* 资源消耗降低 90%+（无需加载页面）
* 稳定性提高（不受页面结构变化影响）
* 检测风险降低（无浏览器指纹）

**建议**：在 `feature/dashboard-api` 合并后，创建 `refactor/task-api-migration` 分支

### 2.2 🟡 P1 - 反检测方案评估

**当前问题**：自实现反检测可能不如成熟库完善。

**Python 项目现状**（`anti_ban_module.py`，~420 行）：

* 10 个反检测脚本
* 覆盖：webdriver 隐藏、plugins、languages、permissions、chrome 对象、platform、hardwareConcurrency、deviceMemory、WebGL 指纹、Canvas 指纹
* ✅ 已覆盖主要场景

**TS 项目方案**：

* `fingerprint-generator` + `fingerprint-injector`
* 专业团队维护，覆盖更全面
* 自动生成一致的浏览器指纹

| 方案                   | 说明                | 成熟度   | 覆盖范围 |
| -------------------- | ----------------- | ----- | ---- |
| fingerprint-injector | TS 项目使用           | ⭐⭐⭐ 高 | 全面   |
| 自实现                  | Python 项目当前方案     | ⭐⭐ 中  | 主要场景 |
| Patchright           | Playwright 反检测增强版 | ⭐⭐⭐ 高 | 全面   |

**建议**：

1. 当前自实现已覆盖主要场景，可继续使用
2. 后续评估 `Patchright`（与 Playwright API 兼容，迁移成本低）
3. `fingerprint-injector` 无 Python 版本，不考虑

### 2.3 🟢 P2 - 鼠标模拟库评估（已验证：无需迁移）

**当前问题**：自实现贝塞尔曲线可能遗漏边界情况。

**Python 项目现状**（`human_behavior_simulator.py`，~320 行）：

* ✅ 三次贝塞尔曲线（Cubic Bezier）
* ✅ Ease-in-out 时间曲线
* ✅ 随机控制点变化
* ✅ 微移动模拟（random_mouse_movement）
* ✅ 思考停顿（10% 概率）

**结论**：实现相当完善，无需迁移到 ghost-cursor。

### 3.2 Fallback 策略

**当前问题**：现有代码无 API fallback 机制，直接迁移风险高。

**推荐实现**：

```python
class UrlRewardTask(Task):
    async def execute(self, page: Page) -> bool:
        # 1. 尝试 API 方式
        if await self._try_api_method():
            return True
        
        # 2. Fallback 到浏览器方式
        self.logger.warning("API 方式失败，回退到浏览器方式")
        return await self._try_browser_method(page)
```

### 3.3 旧代码处理策略

| 策略            | 优点   | 缺点             | 建议     |
| ------------- | ---- | -------------- | ------ |
| 完全删除          | 代码简洁 | 无 fallback，风险高 | ❌ 不推荐  |
| 保留作为 fallback | 稳定性高 | 代码冗余           | ✅ 推荐   |
| 重构为混合模式       | 最优解  | 开发工作量大         | ✅ 长期目标 |

## 四、API 认证验证结果

### 4.1 Quiz API 认证

**端点**：`https://www.bing.com/bingqa/ReportActivity?ajaxreq=1`

* ✅ **只需 Cookie 认证**
* 需要 fingerprint headers
* 请求体：JSON 格式 `{"UserId": null, "TimeZoneOffset": -60, "OfferId": "...", "ActivityCount": 1, "QuestionIndex": "-1"}`
* **可直接迁移，无需额外认证**

### 4.2 URL Reward API 认证

**端点**：`https://rewards.bing.com/api/reportactivity`

* ⚠️ **需要 Cookie + `__RequestVerificationToken`**
* `requestToken` 需要从页面提取（`input[name="__RequestVerificationToken"]` 或 `meta[name="__RequestVerificationToken"]`）
* 还需要 `hash` 和 `offerId`（来自 Dashboard 数据）
* **不能完全脱离浏览器**，至少需要一次页面访问获取 token

### 4.3 迁移复杂度对比

| API            | 认证方式                | 能否完全脱离浏览器  | 迁移复杂度 |
| -------------- | ------------------- | ---------- | ----- |
| Quiz API       | Cookie              | ✅ 可以       | 🟢 低  |
| URL Reward API | Cookie + CSRF Token | ❌ 需要一次页面访问 | 🟡 中  |

**Python 项目现状**：当前没有 `__RequestVerificationToken` 提取逻辑，需要新增。

### 5.1 Fallback 保留策略

**问题**：浏览器方式代码长期保留还是设定过渡期？

**选项**：

* A. 长期保留作为 fallback
* B. 设定 3 个月过渡期后删除
* C. 根据 API 稳定性动态决定

**建议**：选择 A（长期保留），因为：

1. URL Reward API 需要 CSRF Token，不能完全脱离浏览器
2. 保留 fallback 提高系统健壮性

### 5.2 反检测迁移时机

**结论**：

* 当前自实现已覆盖主要场景（10 个反检测脚本）
* 鼠标模拟实现完善（贝塞尔曲线 + ease-in-out）
* 暂不需要迁移，后续可评估 Patchright
